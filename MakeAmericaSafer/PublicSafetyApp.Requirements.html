Public Safety App - Draft Requirements

    The following is a draft set of requirements for a yet to be developed public safety application. It is referred to as a single thing, but is really a series of services which operate on various platforms and utilize many different technologies. The expression of requirements serves the following:

    • Expose things of value which promote public safety.
    • Form a community focused on promoting public safety through improved technology and training.
    • Get the ball rolling on iterative expression and refinement of requirements.
    • Ultimately rollout an agreed upon set of requirements (i.e., baseline process).
    • Utilize requirements to evolve a baselined spec (public interfaces).
    • Encourage rollout of commercial solutions which satisfy requirements and adhere to baselined specification.
Assumptions
  • Site will be fully covered by multiple cameras, all of which contain microphones and speakers.
  • Cameras will be integrated with centralized brain (i.e., Public Safety App).
  • Site infrastructure (e.g., buildings, lighting, doors, elevators, electrical, sound system, etc.) will be integrated with Public Safety App.
  • Security staff will exist to configure and monitor Public Safety App.
  • Local and state regulators, first responders, and IT experts will be intimately involved.
  • Baselined requirements and specifications allow (and mandate) a single view of behavior across diverse sites and locations (i.e., everything from Public Safety App perspective should work exactly the same regardless of particular site or service provider operating security for the site).
  • Because solutions will be implemented using baselined specifications, interoperability across sites and locations will be seamless (facilitates real time sharing of data when appropriate).
Notification Triggers
  • Public Safety App allows for configuration of trigger rules, some examples are:
    • Key word phrases (e.g., gun, shooter, "get down")
    • Certain sounds (gun shot, mass screaming)
    • Black listed individuals
    • Hostile actions (e.g., person exposing a fire arm or other weapon, physical assault)
    • Mass panic (e.g., rapid dispersion of people from a specific location)

  • Public Safety App will continuously monitor configured triggers using a variety of techniques, some examples are:
    • Identification entered into computerized system (e.g., scan of identification badge, person supplies their name to a receptionist)
    • Facial recognition
    • Detection of aggressive action
    • Sound detection (e.g., gun shot)
    • Behavioral anomalies (e.g., someone wearing a heavy coat on a warm summer day, suspicious loitering)

  • Once hostile actor is identified, their movement within the site will be tracked.
  • Public Safety App will fire defined Notification Actions (see below) when configured trigger criteria is detected.
Notification Actions
  • Alarms
  • Notification sent to onsite security and first responders
  • Heightened collection of data from input sources - for example:
    • recent backlog of collection streams (say last 30 minutes) pushed to server
    • higher push rate of real time streams to server

  • Dashboard goes into "crisis mode" (begins to prioritize content based on Public Safety App algorithm)
  • Onsite building infrastructure controlled by Public Safety App (e.g., lights, alarms, auto locking doors, elevator control, etc.)
  • Continuous push of location of hostile actor to smart phones (e.g., within site map)
  • Video and/or still images of hostile actor pushed to smart phones
  • Instructions from onsite security or first responders pushed to smart phones
Video Analysis
  • Continuously scan for:
    • black listed individuals (facial recognition)
    • black listed words or sounds
    • mass panic (rapid dispersion)
    • hostile actions

  • Signal notifications when trigger criteria is met
  • When hostile actor is identified, place appropriate camera feeds in dash board (i.e., camera feeds which show hostile actor)
  • Place best still image of hostile actor in dashboard
  • Continuously adjust dashboard feeds as hostile actor moves throughout the site
  • Visually highlight hostile actor in video feeds (to distinguish from others that are trapped)
  • Prediction of hostile actor location within site (including time estimates) based on prior movement history
  • Hostile action will trigger analysis of historical video streams to look for persons of interest and potential accomplices.
Smart Phones
  • Allow automatic connect to Public Safety App when notified of crisis
  • Allow automatic toggle to silent mode when notified of a crisis (or when hostile actor is in close proximity)
  • Allow bi-directional exchange of text, audio and video streams when connected to Public Safety App
  • Allow interconnect to others trapped in crisis
  • Support real time biometric tracking of user's vital signs when connected to Public Safety App
Dashboard Management
  • Text messages, audio and video will be aggregated to dash board
  • Content will be included in dash board from various sources
    • onsite cameras
    • smart phones
    • broadcast messages from onsite security and/or first responders
    • onsite repository (e.g., site map, critical methods and procedures, evacuation plan)

  • Content in dash board will be prioritized based on algorithm in Public Safety App
  • Dashboard can be used to trigger interaction with user smart phones (those trapped onsite)
  • Dashboard to contain sitemap including each user's location (derived from registered smart phones)
  • Meta data associated with the smart phone will be available from the dash board (including biometric "heartbeat")
  • Dashboard will support replaying of any video stream content(e.g., rewind, fast forward, pause).
  • Dashboard will support manual control of camera settings (e.g., zoom-in / zoom-out, camera direction).
Video Distribution
  • Public Safety App will push text, audio and video streams to centralized server in real time
  • Amount and frequency of data pushed to server will vary based on whether a crisis has been detected
  • Once data is uploaded to centralized server, it can then be distributed to the appropriate users and user groups
Users and Groups
  • Allow preregistration of Users and Groups
  • User roles will be associated with both Users and Groups
  • The group owner serves as the overall administrator of the group
  • System features are available based on user roles
Security
  • Smart phone users can preregister for connectivity to Public Safety App (implies some sort of background check or verification process).
  • Smart phone users can configure profile settings.
  • Access to system features will be constrained based on user roles.
  • Users that aren't preregistered may access Public Safety App (implies extra authentication step).
  • Public Safety App will have security measures to limit exposure of sensitive information to appropriate authorized users.
  • Public Safety App will have security measures to block access by hostile actor.
System Performance
  • System features will be continuously available (at agreed upon performance levels).
  • Configuration settings will control lag time between when video is captured and when it is available for display and/or distribution (e.g., transport frequency / batch size).
  • All system resources will have a synchronized view of time (synchronized to an accurate clock in a single time zone).
  • When displaying video stream content, the original recording time, distribution time, and current (viewing) time will all be available.
  • To minimize lag between video stream collection and viewing, point to point connection will be supported (e.g., first responder will be able to connect directly to onsite cameras).
  • System allows for manual control of camera settings (e.g., zoom in / zoom out, camera direction) - assuming hardware permits such action.
